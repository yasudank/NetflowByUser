# 視野被覆最適化ソルバーの比較 (FoV Optimization Solvers Comparison)

このディレクトリには、観測ポインティング（視野）数を $N$ に設定した際に、ターゲット全体の被覆数（ユニークカバー数）を最大化するための異なるアプローチのソルバー群が格納されています。

---

## 1. 収録ファイル

* **`optimize_hex_fov_milp.py`** (方針A)
  * **候補点の離散化 ＋ Gurobiによる混合整数線形計画法 (MILP)**
  * 事前に生成したグリッド候補点の中から、ターゲットの重複被覆を考慮しつつ全体のカバー率を最大化する $N$ 個の視野の組み合わせを、Gurobiを用いて数理的に厳密解として解きます。
  * ※ガイド星制約（6つのカメラに $\ge 2$ つのガイド星）を満たす視野のみを事前に選択して最適化にかけます。

* **`optimize_hex_fov_local_search.py`** (方針C)
  * **Greedy初期配置 ＋ 座標降下法 (Coordinate Descent) による局所探索**
  * 現行の Greedy法で求めた $N$ 視野の配置からスタートし、各視野の座標（RA, Dec）および回転角（PA）を局所的に微修正（シフト）させて、全体のユニークカバー率が向上するかどうかを勾配不要の山登り法で最適化します。

* **`compare_fov_solvers.py`**
  * **ベンチマーク比較用スクリプト**
  * 現行の「Greedy法」「方針A (MILP)」「方針C (Local Search)」の3つのソルバーを同じターゲットデータセット（COSMOS等）および Gaia カタログを用いて並行実行し、実行時間、合計被覆数、Priority別の被覆数を測定・集計します。
  * また、各ソルバーによる視野の配置結果を視覚的に比較するためのプロット画像（`compare_fov_coverage.png`）を生成します。

---

## 2. ベンチマークの実行方法

仮想環境の Python を用いて、以下のコマンドで実行できます：

```bash
# 4視野 (num-fovs 4) の最適化比較を行う場合
.venv/bin/python fov_solvers_comparison/compare_fov_solvers.py \
    --input ./cosmos/targets_all_20260514.csv \
    --gaia-catalog ./cosmos/gaia.ecsv \
    --num-fovs 4
```

### 主要なオプション
* `--input` : 対象のサイエンスターゲット CSV のパス（デフォルト: `cosmos/targets_all_20260514.csv`）
* `--gaia-catalog` : ガイド星判定用の Gaia ECSV カタログのパス（デフォルト: `cosmos/gaia.ecsv`）
* `--num-fovs` : 最適化する視野数（デフォルト: `4`）
* `--min-stars` : ガイドカメラ1つあたりに最低限必要なガイド星の数（デフォルト: `2`）
* `--min-cams` : 十分なガイド星が存在すべきガイドカメラの最低数（デフォルト: `6`、最大6）

---

## 3. 各アルゴリズムの特性と知見

1. **Greedy法（現行）**:
   * 第1段階で被覆数の大きい座標を粗く探索し、第2段階（Fine Search）で $0.002^\circ$ 刻みの極小グリッドと $0.5^\circ$ 刻みの角度探索によってガイド星制約を満たすよう微調整します。
   * この「第2段階の極小調整」が極めて強力なため、ガイド星制約を維持しつつターゲットカバー数を高く保った高精度な配置を算出できます。

2. **方針A (MILP)**:
   * グリッド候補点に基づく数理最適化を行うため、重複を考慮した最適な組み合わせを厳密に求められます。
   * ただし、ガイド星制約は座標の $0.01^\circ$ 程度の微小な変化に非常に敏感なため、粗いグリッド上では制約を満たさず不採用となってしまう候補が多くなり、結果としてカバー率が Greedy法より低下する傾向があります（極めて細かいグリッドで解く場合は候補数が膨大になり計算時間がかかります）。

3. **方針C (Local Search)**:
   * Greedy法による高精度な初期値から出発し、その周辺で局所最適化（シフト調整）を行います。すでにGreedy側で高度に最適化されているため被覆数がさらに伸びるケースは限定的ですが、実行時間が短く、Greedy結果の最終調整ステップとして有用です。
